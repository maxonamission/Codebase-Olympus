# Story F1-06: content_ref expliciet zetten in alle graph-JSONs

## Doel
Nu is `content_ref` in álle 800 knopen leeg (`grep -c 'content_ref' data/graph/*.json` geeft 0). De loader `_load_scaffolding_content` valt daarom altijd terug op ID-gebaseerde filename-matching (`content_dir / f"{knoop.id}.md"`). Dat werkt, maar:
- Het maakt onzichtbaar welke knopen daadwerkelijk content hebben (zonder directory-listing)
- Hernoemen of verplaatsen van een markdown breekt stilletjes
- De relatie knoop↔content is niet valideerbaar in de bestaande `ValidationReport`

## Input
- `data/graph/*.json`
- `data/content/*.md`
- `src/gymnasium_classica/graph/validation.py`

## Acceptatiecriteria
- [x] Script `scripts/link_content_refs.py`: voor elke knoop met matching `{ID}.md` → `content_ref: "data/content/{ID}.md"` wordt gezet in het juiste graph-JSON-bestand
- [x] Script is idempotent: opnieuw draaien doet niets als alles al correct staat
- [x] Nieuwe validator-check in `graph/validation.py`: voor elke `content_ref` bestaat het bestand op disk; rapporteert ontbrekende of hangende content
- [x] Bestaande 113+ tests groen
- [x] Minstens 36 knopen hebben na de run een `content_ref`
