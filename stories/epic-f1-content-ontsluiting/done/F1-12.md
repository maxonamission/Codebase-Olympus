# Story F1-12: Letterlijk antwoord opslaan + server-side grading

## Doel
De leerling typt een antwoord of kiest een MC-optie, maar de backend ziet
vandaag alleen `correct`/`incorrect`/`slow_correct` — nooit wat er écht
werd ingevuld. Gevolg: een mentor kan hooguit zien *dat* een leerling
worstelt met een knoop, niet *hoe*. En `ItemResponse.item_history` wordt
nergens gevuld — de datastructuur bestaat maar wordt niet geschreven.

Daarnaast: de frontend stuurt voor MC/typed vandaag `response: 'selected'`
of `'typed'`, wat de backend-enum `ResponseType` niet accepteert. Deze
flow faalt dus op 422 — effectief werken alleen de self-assess-knoppen.

## Input
- `src/gymnasium_classica/models/learner.py` — `ItemResponse`, `KnoopState.item_history`
- `src/gymnasium_classica/api/schemas.py` — `AnswerRequest`
- `src/gymnasium_classica/api/session_manager.py` — `submit_answer`
- `src/gymnasium_classica/scheduling/session.py` — `_process_response`
- `frontend/src/api.js`, `Session.jsx`, `AnswerInput.jsx`

## Acceptatiecriteria
- [ ] Nieuwe pure functie `grade_answer(raw, item, taal)` in
      `scheduling/grading.py` — retourneert `GradingResult(correct,
      normalized_answer, normalized_expected)`. Minimale normalisatie:
      `strip`, case-insensitive voor Latijn; NFC + strip-macron voor
      Latijn; NFC voor Grieks. Ondersteunt `item.antwoord: str | list[str]`.
- [ ] `ItemResponse` krijgt optionele velden `answer_text`,
      `correct_answer`, `item_type`. Bestaande velden blijven verplicht
      voor achterwaartse compatibiliteit.
- [ ] `_process_response` hangt een `ItemResponse` aan
      `KnoopState.item_history` (zelfde functie die vandaag niemand
      aanroept — pluspunt voor telemetry).
- [ ] `AnswerRequest` krijgt optioneel `answer_text: str | None`;
      `response` wordt optioneel. Route-logica:
        * `answer_text` aanwezig → server grade't via `grade_answer`,
          `slow_correct` afgeleid uit `response_time_ms >
          1.5 × item.verwachte_tijd_sec * 1000`
        * alleen `response` → zelfbeoordeling, huidig gedrag
        * geen van beide → 422
- [ ] Frontend: `submitAnswer` accepteert
      `{ answerText?, response?, responseTimeMs }`; `Session.handleAnswer`
      splitst op literal-answer vs self-assess.
- [ ] Pytest-tests voor grading (exact match, case, whitespace, macron,
      list-antwoord, Greek NFC), voor `_process_response` (item_history
      wordt gevuld) en een API-integratietest (POST met `answer_text`
      grade't server-side).
- [ ] Bestaande 498 pytest-tests + frontend-tests blijven groen.

## Scope
- Puur Python + Pydantic + bestaande scheduling-pipeline
- Minimale frontend-wijziging
- **Geen** mentor-UI (aparte story)
- **Geen** foutclassificatie / diagnostiek (zie F1-13 of track C)

## Route open naar C
Deze story legt de datavoorraad neer waarop C kan bouwen:

- `answer_text` + `correct_answer` snapshot blijven bewaard — C
  analyseert patronen achteraf zonder migratie.
- `GradingResult` is een frozen dataclass waaraan C velden
  (`mismatch_type`, `confidence`, `hints`) kan toevoegen zonder
  callers te raken.
- `grade_answer(raw, item, taal)` neemt het volledige `Item` mee —
  C kan type-specifieke regels toevoegen (bv. "naamvalsfout"
  herkennen) zonder de API-laag te veranderen.

## Niet-doel
- Geen mentor-dashboard — dat is een eigen story.
- Geen classificatie van foute antwoorden (synoniem-verwarring,
  naamvalsfout, spellingsfout). C-track.
- Geen migratiescript voor bestaande lege `item_history`.
